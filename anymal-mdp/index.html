Understanding MDPs.

Markov Decision Processes give us a mathematical framework for modelling decision-making where outcomes are partly random and partly under the control of a decision maker. Here, that decision maker is ANYmal.

An MDP has five components. We'll visit them one at a time.

States (S).

The set of all possible situations the system can be in.
At any instant, the state captures everything you need to know to make a decision.

In our game

A state is a snapshot of the grid:

  • the position of ANYmal
  • the positions of both ghosts
  • the position of the current reward

Transition function P(s'|s,a).

The probability of landing in state s' after taking action a from state s.
There is randomness in an MDP — the same action in the same state may lead to different outcomes.

In our game

ANYmal occasionally malfunctions and moves randomly (failure probability), and the two ghosts move stochastically. So the next state isn't fully determined by ANYmal's action. Similar to the real world.

Discount factor γ.

A number between 0 and 1 that controls how much we value future rewards versus immediate ones.
A γ close to 1 means the agent cares about long-term consequences; a γ close to 0 makes it short-sighted.

In our game

We use γ = 0.95 — ANYmal is encouraged to plan several steps ahead while still preferring sooner rewards over later ones.

Why bother formalising this with MDPs?

MDPs give us a single mathematical language for three things every reinforcement-learning agent has to handle:

🎲
Uncertainty. Malfunctions, random ghost moves — the world is not deterministic.
🔗
Sequential decisions. One move at a time, each affecting the next.
📈
Long-term outcomes. Maximise total expected reward, not just the next step.

What if we take action when the game is over?

What happens if take_action is called when state.game_over is True?

The very first lines of take_action are a guard:
if state.game_over: return state.copy()
This means once a game has ended, no further mutations happen — the same state is handed back as a defensive copy, so callers can never accidentally resurrect or alter a finished game.

When does ANYmal misbehave?

Under what condition does ANYmal fail to perform the intended action?

This is a classic Bernoulli trial: each call to random.random() returns a uniformly distributed number in [0, 1); comparing it to FAILURE_PROBABILITY turns it into a coin flip with the desired probability. When it succeeds, ANYmal's intended action is replaced with a uniformly random one from Action — that's exactly the malfunction that makes P(s'|s,a) a true probability distribution.

When does a new reward spawn?

Under what condition does take_action call spawn_reward?

Two conditions must both hold for a new reward to spawn: new_reward_pos is None (the previous reward was just collected, setting the position to None) and not game_over. Only one reward exists on the grid at a time, ghost collisions give a large negative reward, and the just-collected reward is removed rather than left in place.

Burn these five into memory — every RL algorithm
you will ever meet is built on top of them:

  • States — a snapshot of the world at each timestep
  • Actions — the choices available to the agent
  • Transitions — the probability of moving from one state to another
  • Rewards — the signal that tells the agent how well it is doing
  • Discount factor — how much the agent values future rewards
